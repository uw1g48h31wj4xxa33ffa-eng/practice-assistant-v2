# 補助金フロー横展開 調査・設計メモ

## 1. 現在の補助金フロー構成

現在の補助金フローは、`workflowTemplates.ts` に `subsidy_v1` として定義されており、以下の6工程で構成されています。

1. **ヒアリング**: `hearing` (共通)
2. **公募要項整理**: `guideline_review` (固有)
3. **必要資料整理**: `document_prep` (固有)
4. **スケジュール管理**: `schedule_management` (固有)
5. **AI検証・エビデンス**: `ai_evidence` (固有)
6. **納品・提出準備**: `delivery_prep` (固有)

**アーキテクチャの特長**
- 各工程は個別の画面（`/cases/[id]/[工程名]/page.tsx`）として独立。
- 各画面は `Case` オブジェクトの特定プロパティ（例: `subsidyGuidelineItems`, `requiredDocuments`）を読み書きする。
- `updateCase` を通じてローカルストレージにデータを永続化。
- 次のステップへの遷移は `progressStatus` を更新した上で、`nextStep.href` へリダイレクトする設計。
- AI生成処理はモック関数 (`mockDeliveryGenerator` など) で代替。

## 2. 共通化できる機能

横展開する際、どの業務フローでも再利用できる共通基盤部分は以下の通りです。

- **ヒアリング画面 (`hearing`)**: 依頼内容や顧客課題をフリーテキスト・メモから抽出する工程はすべての業務で共通。
- **案件管理レイアウト (`/cases/[id]/page.tsx`)**: 全体進捗の可視化、ドーナツチャートによる完了割合の計算、次工程へのガイドUIはそのまま流用可能。
- **UIコンポーネント**: `CompletionActionArea`, `StatusBadge`, `Chip`, `HumanApprovalBadge` などは業務依存しない汎用設計。
- **AI検証の基本思想 (`ai_evidence`)**: 「AIが生成したリストに対して、人間が確認・要修正マークをつける」というUI/UXは、業務を変えても（例：契約書の条項チェック、規程の法違反チェックなど）使い回しが可能。

## 3. 業務固有にすべき機能

横展開する際、新たに実装またはカスタマイズが必要となる部分は以下の通りです。

- **固有プロパティの追加**: `src/types/index.ts` の `Case` 型に、業務専用のデータ保存用配列（例: `grantItems`, `contractClauses`, `visaDocuments`）を追加。
- **要件・ルール整理画面**: 補助金の「公募要項整理」に相当する画面。
  - 契約書なら「契約条件・条項整理」
  - 規程なら「就業規則ヒアリングシート」
- **必要資料整理画面**: 集めるべき書類のリストアップ。業務ごとにモックデータやAI生成のプロンプトを個別に定義する必要がある。
- **提出・納品準備画面**: 補助金のような「締切・電子申請」があるものと、契約書のような「製本・押印」があるものでは、最終確認の項目が変わるため固有画面が必要。
- **`workflowTemplates.ts` への定義追加**: 新しいテンプレートIDと工程リストの登録。

## 4. 横展開しやすい業務候補

補助金と構造が似ている（要件定義 → 資料収集 → スケジュール管理 → 提出準備）業務は横展開が容易です。

**特に横展開しやすい業務（高類似度）**
- **助成金**: 制度要件の確認から申請までのフローが補助金とほぼ一致。
- **行政手続・許認可（建設業許可、古物商許可、法人設立など）**: 要件確認、多数の法定書類の収集、窓口提出というフローが共通。
- **在留資格**: 要件確認、必要書類（本国から取り寄せるものなど）のリストアップ、入管への提出フローが共通。

**やや構造が異なる業務（中〜低類似度）**
- **就業規則・労務相談**: 提出期限や書類収集よりも、条文のカスタマイズや法的リスクのAI検証（エビデンス確認）の比重が高い。
- **契約書**: 相手方との交渉プロセスや条文修正履歴がメインとなるため、バージョン管理的な画面が追加で必要になる可能性がある。

## 5. 優先順位

展開コストが低く、現在の画面コンポーネントを流用しやすい順序です。

1. **助成金** (既存の `subsidy_v1` をほぼそのまま `grant_v1` として流用可能)
2. **許認可・行政手続（建設業許可など）** (要件確認・資料整理の仕組みが活きる)
3. **在留資格** (同上)
4. **法人設立** (タスク管理・スケジュール管理の色合いが強い)
5. **就業規則・契約書** (条文チェック用のUI拡張が必要になるため優先度下)

## 6. 実装時の注意点

- **型の肥大化**: `Case` 型に各業務固有のプロパティ（`xxxItems`）を無秩序に追加すると型が肥大化するため、将来的には `payload: any` や業務ごとの別テーブルに分けるリファクタリングを検討すべき。
- **モックジェネレーターの分離**: `mockRequiredDocumentsGenerator.ts` などが現在補助金にハードコードされているため、引数で業務タイプ（`caseType`）を受け取ってロジックを分岐させるか、ジェネレーター自体を業務別に分ける必要がある。
- **ルーティングの汎用化**: `/cases/[id]/subsidy-schedule` のようなURL直書きを避け、可能であれば `/cases/[id]/[stepId]` のような動的ルーティング側でコンポーネントを出し分ける構造にすると、新規業務追加時のファイル作成の手間が省ける。

## 7. 今後のロードマップ案

1. **フェーズ1（近接業務の展開）**: 助成金テンプレートの追加。既存の補助金画面を一部汎用化し、助成金でも使えるようにコンポーネントを共通化する。
2. **フェーズ2（許認可業務の展開）**: 建設業許可などの許認可テンプレート追加。必要資料整理画面（`required-documents`）を汎用書類収集画面へリファクタリング。
3. **フェーズ3（条文・規程業務の展開）**: 就業規則テンプレートの拡充。条文の差分（Diff）や、法改正チェックに特化した新しいAI検証UIの開発。
4. **フェーズ4（バックエンド結合）**: `localStorage` からクラウドDBへの移行と、実際のLLM API（OpenAI等）との接続によるモックからの脱却。
